## 1. Scope

v0.2.0 adds four capabilities to model-ledger:

| Feature | Repo | Description |
|---|---|---|
| Introspection protocol + plugin system | OSS | Typed protocol for extracting metadata from model objects and configs |
| Reference introspectors | OSS | sklearn, XGBoost, LightGBM |
| CLI | OSS | Typer-based CLI for list, show, validate, introspect, audit-log, export |
| Audit pack export | OSS | HTML/JSON/Markdown export of model card + validation + audit trail |

Block-specific work (in `forge-block-mrm/projects/model_ledger_block`):

| Feature | Description |
|---|---|
| SquareWave/Snoop introspector | Extracts heuristic queue metadata from SquareWave configs |
| Gondola introspector | Extracts Risk ML model metadata |
| Snowflake backend | `InventoryBackend` implementation backed by Snowflake |

**Explicitly out of scope for v0.2.0:** Excel import/export, PDF export, SQLAlchemy/Postgres backend, model comparison/diffing.

## 2. Introspection Protocol

### 2.1 Protocol Definition

```python
from typing import Any, Protocol

class Introspector(Protocol):
    """Extracts metadata from a model object or config."""
    name: str

    def can_handle(self, obj: Any) -> bool: ...
    def introspect(self, obj: Any) -> IntrospectionResult: ...
```

All introspectors implement this protocol. `can_handle` returns True if the introspector knows how to extract metadata from the given object. `introspect` returns a typed `IntrospectionResult`.

**Exceptions:** `NoIntrospectorError` is added to `model_ledger.core.exceptions`. It subclasses `ModelLedgerError` and takes a `target` argument (the type or name that couldn't be matched):

```python
class NoIntrospectorError(ModelLedgerError):
    def __init__(self, target: type | str) -> None:
        self.target = target
        label = target.__name__ if isinstance(target, type) else target
        super().__init__(f"No introspector found for {label}")
```

### 2.2 Plugin Discovery

Introspectors register via Python entry points:

```toml
# In a plugin's pyproject.toml
[project.entry-points."model_ledger.introspectors"]
sklearn = "model_ledger.introspect.sklearn:SklearnIntrospector"
```

The SDK maintains a registry that auto-discovers installed plugins via `importlib.metadata.entry_points()`. Discovery is lazy — no import cost until someone calls `introspect()`.

Manual registration is also supported:

```python
from model_ledger import register_introspector
register_introspector(MyCustomIntrospector())
```

### 2.3 Registry

```python
class IntrospectorRegistry:
    def __init__(self):
        self._introspectors: list[Introspector] = []
        self._discovered = False

    def register(self, introspector: Introspector) -> None:
        """Register an introspector. Deduplicates by name — later registration wins.
        Prepends so manually registered introspectors take priority over entry-point-discovered ones."""
        self._introspectors = [i for i in self._introspectors if i.name != introspector.name]
        self._introspectors.insert(0, introspector)

    def find(self, obj: Any) -> Introspector:
        """Find first introspector that can handle obj."""
        self._ensure_discovered()
        for intro in self._introspectors:
            if intro.can_handle(obj):
                return intro
        raise NoIntrospectorError(type(obj))

    def get_by_name(self, name: str) -> Introspector:
        """Get introspector by name."""
        self._ensure_discovered()
        for intro in self._introspectors:
            if intro.name == name:
                return intro
        raise NoIntrospectorError(name)

    def _ensure_discovered(self) -> None:
        """Lazy-load entry points on first use."""
        if not self._discovered:
            # load from importlib.metadata.entry_points("model_ledger.introspectors")
            self._discovered = True
```

## 3. IntrospectionResult Data Model

The typed contract that all introspectors return. Handles both ML models and heuristic/config models.

**Module location:** All types in this section live in `model_ledger/introspect/models.py`. They are re-exported from `model_ledger.introspect`. `ComponentInfo` is a DTO for introspection results only — it maps to `ComponentNode` (in `core/models.py`) when applied via `_apply_result`, but they are separate types.

### 3.1 Supporting Types

```python
class FeatureInfo(BaseModel):
    name: str
    dtype: str | None = None          # "float64", "int", "string", "embedding"
    source: str | None = None         # "dumbo", "snowflake_feature_engine", "raw"
    description: str | None = None
    metadata: dict[str, Any] = {}

class DataSourceInfo(BaseModel):
    name: str                         # table name, file path, API endpoint
    source_type: str                  # "snowflake_table", "csv", "api", "feature_store"
    fields: list[str] = []            # columns/fields used
    filters: dict[str, Any] = {}      # WHERE clauses, lookback windows
    description: str | None = None
    metadata: dict[str, Any] = {}

class ThresholdInfo(BaseModel):
    name: str                         # "model_score", "total_in_amount"
    value: float | str
    segment: str | None = None        # "high_risk", "regular_1"
    operator: str | None = None       # ">=", "<=", "=="
    description: str | None = None

class ComponentInfo(BaseModel):
    """Maps to a ComponentNode in the version's tree."""
    path: str                         # "Inputs/features", "Processing/model"
    node_type: str                    # "feature_set", "algorithm", "threshold_set"
    metadata: dict[str, Any] = {}
```

### 3.2 IntrospectionResult

```python
class IntrospectionResult(BaseModel):
    # Identity
    introspector: str                 # "sklearn", "squarewave"
    framework: str | None = None      # "scikit-learn", "xgboost", "squarewave"

    # Model architecture
    algorithm: str | None = None      # "LGBMClassifier", "heuristic_rule"
    hyperparameters: dict[str, Any] = {}

    # Inputs
    features: list[FeatureInfo] = []
    data_sources: list[DataSourceInfo] = []

    # Outputs & decision logic
    thresholds: list[ThresholdInfo] = []
    metrics: dict[str, float] = {}    # "precision": 0.967
    output_description: str | None = None

    # Operational
    execution_schedule: str | None = None   # "daily", "twice_monthly"
    lookback_window: str | None = None      # "30 days"

    # Structure mapping
    components: list[ComponentInfo] = []

    # Escape hatch
    metadata: dict[str, Any] = {}
```

### 3.3 Example Mappings

**ML model (RTB Drug Sales, model 0040):**
- `algorithm`: "LGBMClassifier"
- `features`: 1041 entries (BERT embeddings, Dumbo features, Snowflake Feature Engine)
- `metrics`: {"precision": 0.967, "threshold": 0.83}
- `thresholds`: [ThresholdInfo(name="model_score", value=0.83, operator=">=")]
- `execution_schedule`: "real_time"

**Heuristic (TM Many-to-One from rulebook):**
- `algorithm`: "heuristic_rule"
- `data_sources`: [DataSourceInfo(name="app_compliance.cash.queue_execution_metrics_p2p_archive", source_type="snowflake_table", filters={"lookback": 30, "currency_code": "USD"})]
- `thresholds`: [ThresholdInfo(name="total_in_amount", value=2000, segment="regular_1", operator=">="), ...]
- `execution_schedule`: "twice_monthly"
- `lookback_window`: "30 days"

Block-specific fields (snoop_algorithm_id, squarewave_job_id, queue_name) go in the `metadata` escape hatch.

## 4. SDK Integration

### 4.1 DraftVersion Changes

`DraftVersion` gains one public method:

```python
def introspect(self, obj: Any, *, introspector: str | None = None) -> IntrospectionResult:
    """Introspect obj and apply results to this version."""
    # Lazy import to avoid circular dependency (model_ledger -> sdk -> model_ledger)
    from model_ledger.introspect.registry import get_registry
    registry = get_registry()
    if introspector:
        intro = registry.get_by_name(introspector)
    else:
        intro = registry.find(obj)
    result = intro.introspect(obj)
    self._apply_result(result)
    return result

def _apply_result(self, result: IntrospectionResult) -> None:
    """Map IntrospectionResult onto version fields and component tree.

    This is best-effort — all fields are applied independently. The raw
    IntrospectionResult is also stored in version metadata for audit pack
    reconstruction and debugging.
    """
    if result.algorithm:
        self._version.methodology_approach = result.algorithm
    if result.features:
        self.add_component("Inputs/features", type="feature_set",
            metadata={"count": len(result.features),
                       "features": [f.model_dump() for f in result.features]})
    if result.thresholds:
        self.add_component("Processing/thresholds", type="threshold_set",
            metadata={"thresholds": [t.model_dump() for t in result.thresholds]})
    if result.data_sources:
        for ds in result.data_sources:
            self.add_component(f"Inputs/{ds.name}", type="data_source",
                metadata=ds.model_dump())
    if result.execution_schedule:
        self.set_run_frequency(result.execution_schedule)
    # Store raw result for audit pack export
    self._version.artifacts.append(
        ModelArtifact(artifact_type="introspection_result",
                      uri=f"introspect://{result.introspector}",
                      checksum=None))
    # ... remaining field mappings
```

**Idempotency:** If `introspect()` is called multiple times on the same draft, `_apply_result` appends new components without deduplicating. This is intentional for v0.2.0 — the user controls this by calling introspect once. Future versions may add upsert semantics to `add_component`.

**Save-on-exception:** The existing `DraftVersion.__exit__` saves unconditionally. This is preserved — `_apply_result` applies fields independently, so a partial result (e.g., algorithm set but features missing due to a later error) is still useful context. The audit trail records the introspection attempt.

### 4.2 Standalone Function

```python
# model_ledger/__init__.py
def introspect(obj: Any, *, introspector: str | None = None) -> IntrospectionResult:
    """Introspect a model object or config without an inventory session."""
    registry = get_registry()
    if introspector:
        intro = registry.get_by_name(introspector)
    else:
        intro = registry.find(obj)
    return intro.introspect(obj)
```

### 4.3 User Override Pattern

```python
with inv.new_version("arr-v3") as v:
    result = v.introspect(fitted_model)
    # Override what introspection found
    v.set_training_target("SAR filing within 90 days")
    v.set_run_frequency("twice_monthly")
```

## 5. Reference Introspectors

Three ML introspectors ship as optional dependencies in the OSS repo.

### 5.1 Introspector Priority

The registry uses **first-match wins** ordering. To avoid ambiguity (e.g., `XGBClassifier` inherits from sklearn's `BaseEstimator`), each introspector's `can_handle` checks concrete types rather than duck-typing:

- **XGBoost** and **LightGBM** introspectors check `isinstance` against their library types
- **sklearn** is the fallback — checks `isinstance(obj, sklearn.base.BaseEstimator)`
- Entry point discovery loads in alphabetical order; manually registered introspectors take precedence (prepended)

The reference introspectors are registered via entry points in the OSS repo's own `pyproject.toml`:

```toml
[project.entry-points."model_ledger.introspectors"]
lightgbm = "model_ledger.introspect.lightgbm:LightGBMIntrospector"
sklearn = "model_ledger.introspect.sklearn:SklearnIntrospector"
xgboost = "model_ledger.introspect.xgboost:XGBoostIntrospector"
```

### 5.6 Pipeline Support

sklearn `Pipeline` objects decompose into components:


## 6. CLI

Typer-based CLI. Rich-formatted output by default, `--format json` for scripting.

### 6.1 Commands

```bash
# Inventory
model-ledger list                                    # list all models
model-ledger show <model>                            # model details + versions
model-ledger show <model> --version <ver>            # specific version

# Validation
model-ledger validate <model>                        # SR 11-7 against latest
model-ledger validate <model> --version <ver> --profile sr_11_7
model-ledger validate <model> --format json          # for CI (exit code 1 on failure)

# Introspection (from serialized artifacts)
model-ledger introspect model.pkl
model-ledger introspect config.yaml --introspector squarewave

# Audit
model-ledger audit-log <model>
model-ledger audit-log <model> --version <ver>

# Export
model-ledger export <model> --format html
model-ledger export <model> --format json
model-ledger export <model> --format markdown
```

### 6.2 Design Decisions

- Default backend: SQLite at `./inventory.db` (override via `--db` or `MODEL_LEDGER_DB` env var)
- `validate` returns exit code 1 on failures for CI integration
- `introspect` from CLI loads serialized artifacts (joblib, ONNX). Pickle files require an explicit `--allow-pickle` flag due to arbitrary code execution risk. A warning is printed when `--allow-pickle` is used.
- Rich tables and formatting by default, `--format json` for machine consumption
- Default export format: HTML (the primary format per Section 7.2)

## 7. Audit Pack Export

Self-contained compliance artifact for MRM teams and auditors.

### 7.1 Contents

- Model card (identity, purpose, owner, tier, risk rating)
- Version detail (algorithm, features, data sources, thresholds, deployment)
- Component tree (Inputs → Processing → Outputs)
- Validation results (profile pass/fail with findings)
- Observations and feedback history (if available — the `InventoryBackend` protocol does not yet include observation storage methods; the `InMemoryBackend` supports them but the protocol needs extending in a future version. For v0.2.0, observation data is included only when the backend provides it.)
- Full audit trail (every mutation, who, when, why)
- Introspection snapshot (reconstructed from the `introspection_result` artifact stored on the version and the mapped fields)

### 7.2 Formats

| Format | Use case |
|---|---|
| HTML | Single-file, self-contained, inline CSS. Share via email/Slack. Primary format. |
| JSON | Machine-readable, diffable, feed into other tools |
| Markdown | Version-controllable, embed in repos or wikis |

No PDF for v0.2.0 (requires heavy native dependencies like weasyprint). HTML prints to PDF.

### 7.3 API

```python
from model_ledger.export import export_audit_pack

export_audit_pack(
    inventory=inv,
    model_name="arr-v3",
    version="0.1.0",
    format="html",
    output_path="arr-v3-audit-pack.html",
)
```

## 8. Project Setup & Boundary Rules

### 8.1 Repository Relationship

```
model-ledger (OSS, ~/model-ledger)
├── Schema, SDK, validation engine, backend protocol
├── Introspection protocol + reference introspectors
├── CLI and export module
└── Apache-2.0, no Block-specific code

model-ledger-block (internal, ~/Desktop/forge-block-mrm/projects/model_ledger_block)
├── Adapters (Yields, Jira, GDrive, Snowflake)
├── Block-specific introspectors (SquareWave, Snoop, Gondola)
├── Snowflake backend
├── Prefect flows (sync, drift detection)
└── Depends on model-ledger via git (or local editable for dev)
```

### 8.2 Local Development Setup

For concurrent development, install model-ledger as an editable dependency:

```bash
cd ~/Desktop/forge-block-mrm/projects/model_ledger_block
uv pip install -e ~/model-ledger
```

### 8.3 Boundary Rule

**"If it needs a Block import, it goes in forge. If it's useful to any org, it goes in OSS. When in doubt, OSS first."**

- Introspection protocol, result types, reference introspectors → OSS
- SquareWave/Snoop/Gondola introspectors → forge
- Snowflake backend → forge
- If a new protocol method or result field is needed, add it to OSS first, then consume in forge

Each repo gets a CLAUDE.md with build commands, architecture overview, and the boundary rules. See Section 8.3 for the governing rule.
