## 1. The Problem Everyone Has

Every bank's model inventory is a spreadsheet. SR 11-7 requires comprehensive model inventories; the industry responds with Excel and SharePoint. Commercial tools (Yields.io, ValidMind, SAS) exist but are expensive, proprietary, and not developer-friendly. No open standard exists.

## 2. Why Now: The Agent Era Changes Everything

AI agents are governing models — AutoValidator proves it. But agents can't consume spreadsheets or PDFs. The gap isn't "we need a better UI" — it's "we need machine-readable governance infrastructure."

The history of AI shows that general methods leveraging computation always win over hand-encoded human knowledge (Sutton, "The Bitter Lesson," 2019). Model governance is about to learn the same lesson. Today's governance is manual: humans assemble context, humans check compliance, humans write reports. model-ledger doesn't try to encode governance intelligence — it builds the structured representation that lets AI agents do the governing. The validation rules are the floor (regulatory minimums), not the ceiling. The real intelligence lives in agents that traverse, reason over, and act on the structured data.

Key distinction: model-ledger is infrastructure (representation), not reasoning. Like how good data formats (protobuf, JSON-LD) enable computation to scale, model-ledger is the governance data format that makes the Bitter Lesson possible for MRM.

## 3. What model-ledger Is

model-ledger is an open-source Python library that provides a formal, machine-readable inventory for model governance. It tracks four first-class entities:

- **Models** — hierarchical I/P/O components per SR 11-7, with versions, ownership, lifecycle states
- **Governance documents** — linked evidence (model specs, validation reports, approval artifacts)
- **Observations** — validation findings from any source (human reviewers, AI agents, automated tools, manual entry), each with full lifecycle tracking
- **Feedback** — structured records of what happened to each observation: kept, removed, modified, and why

A validation engine runs compliance checks. An export layer produces audit packs, gap reports, and agent-consumable configs.

## 4. The Feedback Loop: How Governance Gets Better Over Time

Today, validation observations — whether generated by humans or AI agents — get triaged in spreadsheets. The corrections are never captured structurally. Every validation cycle starts from zero.

model-ledger treats observations as inventory objects with a lifecycle:

> **Observation created** (source: human reviewer, AI agent, tool, etc.)
> → **triaged** (kept / removed / modified — with reason and rationale)
> → **issued** (published in a ValidationReport) or **removed** (preserved in history)

The full history is preserved — every run, every triage decision — but only "issued" observations appear in the final validation report. Like model versions (draft → published), observations have a clear moment when they become official.

Over validation cycles and across models, the triage history accumulates into a structured dataset. Any tool — AI or otherwise — can query this before generating new observations.

**The schema provides stable structure that regulators require. The feedback layer provides the learning surface that agents need. Stability where governance demands it; adaptability where computation can improve it.**

### Internal (model-ledger-block)

## 6. How This Fits with Yields

Yields is the system of record. model-ledger is the standardization and interop layer — it makes Yields data portable, validatable, and agent-consumable. Maps to Phase 7 of the Yields PRD. Complements, never competes.

## 7. How This Fits with Validation Tools

model-ledger is the inventory. Validation tools (AI agents, human reviewers, future tools) are the producers and consumers.

- **Any tool → model-ledger**: Observations are written with a source tag and full lifecycle tracking. The inventory doesn't care who generated the observation.
- **model-ledger → Any tool**: The inventory provides model context (component tree, governance docs, prior observations + feedback history) that tools consume to generate better findings.
- **Neither replaces the other**: model-ledger doesn't generate findings. Validation tools don't store governance data. Clean boundary.

At Block, AutoValidator is the primary AI validation tool — so the internal adapter has a specific integration. But the OSS schema is tool-agnostic by design.
